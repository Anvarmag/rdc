# ТЗ: Конфигуратор услуги + заявка + оплата + интеграции (РДЦ)

Версия: 1.0  
Проект: РДЦ (статический сайт: HTML + Vanilla JS + Tailwind CDN + Python build.py)  
Ограничения: Никакого Node.js, сборщиков, package.json. Все правки - через исходные HTML (ru) + build.py для генерации языков.

## 1. Цель
Сделать новую страницу сайта с конфигуратором услуг (калькулятором) и формой заявки:
- Пользователь выбирает параметры услуги (тип исследования, срок, ИИ, консилиум).
- Система рассчитывает итог, показывает цену.
- Пользователь заполняет анкету + вставляет ссылку на архив исследования.
- Пользователь оплачивает услугу через эквайринг Ак Барс.
- Заявка (лид/сделка) создается в Bitrix24 и обновляется статусом оплаты.

## 2. Страница и маршрутизация
Добавить новую публичную страницу, например:
- /service-configurator.html (RU исходник)
- Генерируемые: service-configurator-en.html, service-configurator-tt.html (build.py)

Страница должна:
- Использовать общую стилистику сайта (шрифты, цвета, отступы, компоненты).
- Иметь общий header/footer как на остальных страницах проекта.
- Быть адаптивной (320..1920+).

## 3. UI/UX требования (Desktop)
### 3.1 Основной блок конфигуратора
Конфигуратор состоит из 4 логических блоков:
1) Виды услуг
2) Результат (срок)
3) Чтение снимка с применением ИИ (опционально, зависит от услуги)
4) Консилиум врачей (опционально)

После выбора параметров всегда показывать блок "Итог" (текстом, списком) + "Цена" + CTA "Перейти к анкете" или "Оплатить" (по логике ниже).

### 3.2 Компоненты выбора
- Все выборы реализовать как кнопки/плитки (как на эскизах).
- Выбранный вариант имеет состояние active.
- До выбора обязательного шага - последующие шаги не активны/скрыты.

### 3.3 Итог
Итог - список:
- Тип услуги
- Срок результата
- Чтение ИИ: да/нет (+ выбранный подтип если да)
- Консилиум: да/нет
- Цена: ХХХ ₽

## 4. Логика конфигуратора

### 4.1 Тип услуги (обязателен, 1 из)
- Описание МРТ исследования (1 область)
- Описание РКТ исследования (1 область)
- Описание рентгеновского исследования (1 область)
- Описание маммографии (1 исследование)
- Описание флюорографии (1 исследование)

### 4.2 Срок результата (обязателен, 1 из)
- в течение 24-х часов
- в течение 3-х часов

### 4.3 Чтение снимка с применением ИИ (опционально)
Показывать блок ИИ только для услуг, где доступно ИИ (см. 6 Прайс).
- Если блок ИИ показан: выбор "Да/Нет" или выбор подтипа и отдельное "Да/Нет" - допустимо, но итог должен содержать "ИИ: да/нет" и при да - выбранный подтип.
- Если блок ИИ не применим: скрывать его полностью, в итог писать "ИИ: нет".

Подтипы ИИ (по эскизам):
- РКТ органов грудной полости и легких
- РКТ головного мозга
- Рентген легких
- Маммография
- Флюорография

### 4.4 Консилиум врачей (опционально)
Текст блока:
"Консилиум врачей (заключение с учетом мнений не менее 3х врачей, включая кандидата медицинских наук)"
Выбор: Да/Нет.

### 4.5 Правило совместимости консилиума со сроком (важно)
По прайсу консилиум указан отдельной стоимостью и без срочного срока.
Принять правило:
- Если Консилиум = Да -> срок результата фиксируется как "24 часа" и "3 часа" недоступен (disabled/скрыт).
- Если Консилиум = Нет -> доступны оба срока (24ч/3ч).

### 4.6 Формула цены
1) Если Консилиум = Нет:
   Цена = Цена(услуга, срок) + (ИИ выбран? ЦенаИИ(услуга или подтип): 0)
2) Если Консилиум = Да:
   Цена = ЦенаКонсилиум(услуга) + (ИИ выбран? ЦенаИИ: 0)
   Срок = 24 часа (принудительно)

Все цены хранятся в конфиге (JSON-объект в js файле), без хардкода в разметке.

## 5. Анкета (форма заявки)

### 5.1 Поля анкеты
Обязательность уточнить, но по умолчанию:
- ФИО (обяз.)
- Телефон (обяз.)
- E-mail (обяз.)
- Город (опц.)
- Возраст (опц.)
- Пол (обяз.: Муж/Жен)
- Вид и область исследования (обяз.)
- Онкозаболевания (опц.)
- Хирургические вмешательства (опц. текст)
- Ссылка на исследование (обяз.) - ссылка на облако
- Жалобы/описание (опц. текст)

### 5.2 Инструкция по загрузке исследования (на странице рядом с формой)
Показать блок инструкции:
- Открыть диск с МРТ/КТ/Рентген
- Сложить в одну папку и создать ZIP
- Загрузить на облако: Яндекс.Диск, Google Drive, Mail.ru Cloud
- Вставить ссылку в анкету
- Заполнить анкету максимально подробно
- Оплатить выбранную услугу
- Менеджер может связаться для уточнения
- Срок результата - согласно выбранной опции
- Контакт для отзыва: info@online-mrt.ru

### 5.3 Порядок обработки заявок
Текст:
- Заявки принимаются круглосуточно
- Оплаченные заказы обрабатываются в рабочее время 09:00-21:00
- Оплаченные после 21:00 передаются врачам в 09:00 следующего дня

## 8. Интеграция Bitrix24
### 8.1 Требование
После оплаты заявка должна появиться в Bitrix24 и иметь статус оплаты.

### 8.2 Безопасность
Нельзя хранить секреты Bitrix24 в браузере.
Нужен серверный endpoint (backend) на текущем сервере (PHP или Python), который:
- принимает JSON формы
- создает лид/сделку через Bitrix24 webhook/REST
- возвращает идентификатор сущности (leadId/dealId)

### 8.3 Отправляемые поля в Bitrix24
- Контакты: ФИО, телефон, email, город
- Возраст, пол
- Вид и область исследования
- Ссылка на архив
- Жалобы/описание
- Выбор конфигуратора: услуга, срок, ИИ, консилиум
- Итоговая цена
- Статус оплаты (Не оплачено / Оплачено)
- ID платежа / orderId

## 9. Эквайринг Ак Барс
### 9.1 Требование
Оплата выбранной услуги через эквайринг Ак Барс.

### 9.2 Архитектура
Нужен backend, который:
1) Создает заявку в Bitrix24 (или подготавливает сущность)
2) Создает платеж (order) в Ак Барс (сумма, назначение, orderId)
3) Возвращает на фронт ссылку/форму оплаты
4) Принимает callback/webhook от банка
5) По callback обновляет Bitrix24: "Оплачено" + сохраняет paymentId/status

### 9.3 Пользовательский сценарий
- Пользователь заполнил конфиг + анкету
- Нажал "Оплатить"
- Перешел на оплату
- После оплаты видит страницу "Спасибо" + что делать дальше
- В Bitrix24 заявка отмечена как оплаченная

## 11. Требования к качеству
- Без внешних тяжелых библиотек.
- JS модульный, читаемый, без дублирования.
- Валидация формы (минимум: обязательные поля + формат email + телефон).
- Ошибки оплаты/интеграций - понятные сообщения пользователю.
- Логи на сервере для callback (минимум).

## 12. Критерии приемки (Acceptance)
- На desktop и mobile пользователь проходит сценарий от выбора до оплаты.
- Цена считается корректно по прайсу и правилам консилиума.
- Если консилиум=да, срок 3ч недоступен.
- Если услуга не поддерживает ИИ, блок ИИ не показывается.
- Заявка появляется в Bitrix24 с заполненными полями.
- После успешной оплаты Bitrix24 получает статус "Оплачено".
- Никакие секреты Bitrix24/банка не лежат в фронтенде.
